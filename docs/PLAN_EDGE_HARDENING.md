# Plan — Edge Hardening: Retire the Blanket Core Proxy

Work order for `archura-editor` (Worker + scripts + docs) — my side of the
security fixes agreed 2026-07-20. Core companion (small, for the other model):
inner auth on machine-invoked billing endpoints — specified in
`core/PLAN_EDGE_HARDENING.md`. No product-behavior changes; this closes the
vulnerability class found in review (any browser could borrow the Worker's
service key for arbitrary `/v1/*` calls via `/api/core/*`, exposing the
unauthenticated site-release endpoint to the public internet).

## Context (settled in discussion, 2026-07-20)

- **Topology stands:** browsers ↔ Worker (BFF, cookies) · Worker ↔ core
  (service key + per-request principal auth) · nobody ↔ core directly except
  `/stripe/webhooks` (self-authenticating via HMAC). The Hetzner firewall
  (Cloudflare-IPs-only on 443) is now applied and verified.
- **Doctrine:** the service header is transport proof only — *every* core
  endpoint self-defends with principal auth (session, membership, signature,
  or internal key). Reachability is never authorization.
- The blanket `/api/core/*` forward has **no product consumers** — every
  surface uses purpose-built BFF routes. Its only users are
  `register-test-client.mjs` and `verify-core-proxy.mjs`.

## Work items

### 1. Remove the blanket proxy from production paths

In `workers/site-worker.js`:

- Delete the unconditional `proxyCore` dispatch for `/api/core/*`. Keep the
  `coreRequest` helper — the BFF routes are the supported path.
- **Dev-only exception:** allow `/api/core/*` forwarding only when the same
  dev gate as the mailbox applies (core reports dev / `PUBLIC_ORIGIN` set
  locally — reuse the existing convention, one condition, commented). In
  production the route returns 404 unconditionally.
- Remove `coreRateLimitOperation` if it becomes dead code with the proxy.

*Verify:* in a prod-shaped run (no dev gate), `POST /api/core/v1/anything` →
404; locally the dev scripts still work.

### 2. Repoint the two dev consumers

- `register-test-client.mjs`: call the local core directly
  (`CORE_URL`/`localhost:8080`) with the admin key + service key it already
  has via env, instead of the Worker proxy. It's a dev tool; it can speak to
  the dev core like the verify suites do.
- `verify-core-proxy.mjs`: retire or repurpose — its job becomes asserting
  the *absence* of the proxy in prod mode and its presence in dev mode
  (rename to reflect that if kept).

*Verify:* `register-test-client.mjs` works against `dev-up.sh`'s stack;
`verify-all` stays green.

### 3. Regression test for the vulnerability itself

Add to the appropriate suite (worker-level, no live core needed — follow the
`verify-worker-billing.mjs` in-memory pattern):

- `DELETE /api/core/v1/organizations/<uuid>/sites/<sub>` through the Worker →
  404 (prod mode), and the request **never reaches** the fake core (assert no
  outbound call).
- The service header cannot be smuggled: a browser request carrying its own
  `X-Archura-Service-Authorization` to any BFF route must have it stripped
  before forwarding (the header-delete already exists in `coreRequest`
  callers — pin it with a test).

### 4. Untrack `.dev.vars`

`git rm --cached archura-editor/.dev.vars` + add to `archura-editor/
.gitignore`. It's regenerated by `dev-up.sh` each run and holds dev keys;
tracking it dirties every commit and normalizes committing key material.

*Verify:* `git status` clean after a `dev-up.sh` run.

### 5. Record the doctrine (docs)

In `docs/AUTH_ARCHITECTURE.md`, a short decided-posture section:

- Every core endpoint self-defends; transport (service header, firewall,
  Cloudflare-only origin) is defense-in-depth, never authorization.
- The two doors: `/v1` (service key + per-request principal auth, Worker-only)
  and `/stripe/webhooks` (HMAC per delivery, public by design).
- Core is not a public API today; when merchant backends with `sk_` become
  real (component-session minting), a deliberate, versioned public surface
  (`api.archura.ai`) is opened for exactly those routes — public-by-decision,
  not by forward rule.
- Test convention: authorization tests must pass with edge auth disabled.

### 6. Full regression

`npm run verify:all` green; `verify-funnel` (29 checks), account suites, and
billing worker suite unchanged. Nothing in this plan touches product flows —
any suite that notices the change (other than the new tests) is a bug in the
plan.

## Explicitly out of scope

- Core-side inner auth for `release`/`start-trial`/`entitlement` — the core
  companion plan (other model).
- The Hetzner firewall and Stripe/email prod setup — operator tasks, done or
  scheduled by Igor.
- Any change to BFF routes, webhook handling, or the embed/identity work.
